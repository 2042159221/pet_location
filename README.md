# 01_Pet_Locator

基于 `STM32F103C8T6` 的宠物定位终端固件工程。项目当前采用 `STM32CubeMX + HAL` 生成底座，在此基础上叠加了 GPS、计步器、NB-IoT 通信模块的业务驱动与调试能力。

当前仓库不是“成品固件”，而是处于联调和问题定位阶段的嵌入式工程。尤其是串口调试、系统时钟初始化、QS100 建链与上传链路，仍然是当前重点关注对象。

## 1. 项目概览

### 1.1 目标能力

- 采集 GPS 定位信息
- 采集步数信息
- 通过 NB-IoT 模块将数据上传到远端服务器
- 通过 `USART1` 输出调试日志，辅助现场排障

### 1.2 核心硬件

| 模块 | 型号 / 说明 | 当前用途 |
| --- | --- | --- |
| MCU | `STM32F103C8T6` | 主控 |
| GPS | `AT6558R` | 获取定位数据 |
| 计步器 | `DS3553` | 获取步数数据 |
| 通信模块 | `QS100` | NB-IoT 网络接入与数据上传 |

## 3. 硬件接口映射

### 3.1 串口分配

| 外设 | 引脚 | 波特率 | 用途 |
| --- | --- | --- | --- |
| `USART1` | `PA9 / PA10` | `115200` | 主调试口，`printf` 重定向输出 |
| `USART2` | `PA2 / PA3` | `115200` | 与 `AT6558R` GPS 模块通信 |
| `USART3` | `PB10 / PB11` | `9600` | 与 `QS100` 模块通信 |

### 3.2 GPIO 分配

| 引脚 | 名称 | 方向 | 用途 |
| --- | --- | --- | --- |
| `PB13` | `QS100_WKUP` | 输出 | 唤醒 QS100 |
| `PB3` | `GPS_EN` | 输出 | 控制 GPS 模块使能 |
| `PB5` | `DS3553_CS` | 输出 | 计步器片选 |
| `PB6 / PB7` | `I2C1_SCL / I2C1_SDA` | 复用 | 与 `DS3553` 通信 |

### 3.3 时钟相关

- 系统时钟当前依赖 `HSE` 外部高速时钟。
- `SystemClock_Config()` 中同时开启了 `HSE + LSI`。
- 若 `HSE` 初始化失败，程序会在进入常规 `printf` 调试输出前直接掉进 `Error_Handler()`。

这不是细节问题，而是当前串口“完全无输出”排障时必须优先怀疑的根因之一。

## 4. 软件结构设计

- `Core/`
  - CubeMX 生成的 HAL 工程入口与底层初始化代码
  - 包含 `main.c`、时钟、GPIO、I2C、RTC、USART、中断处理
- `Com/`
  - 通用能力层
  - 当前包括延时、调试输出、数据结构定义
- `Int/`
  - 外设/模块集成层
  - `AT6558R`、`DS3553`、`QS100` 的驱动与协议封装
- `App/`
  - 业务流程层
  - 负责初始化和采集上传流程编排
- `MDK-ARM/`
  - Keil 工程文件与构建产物

### 4.1 当前模块职责

- `Com_Debug`
  - 通过重写 `fputc()`，将 `printf` 重定向到 `USART1`
- `Com_Delay`
  - 提供毫秒级延时封装
- `Int_AT6558R`
  - 负责 GPS 模块上电、命令发送、串口空闲中断接收
- `Int_DS3553`
  - 负责计步器寄存器读写与步数读取
- `Int_QS100`
  - 负责 AT 链路、网络附着、Socket 创建、服务器连接、数据发送与发送状态确认
- `App_Main`
  - 预留业务初始化与“采集并上传”主流程封装

## 6. 当前程序行为

以当前 `Core/Src/main.c` 为准，上电后的主流程是：

1. `HAL_Init()`
2. `SystemClock_Config()`
3. 初始化 `GPIO / RTC / USART1 / I2C1 / USART2 / USART3`
4. 执行 `Int_QS100_Init()`
5. 直接调用：

```c
Int_QS100_UploadData("8.135.10.183", 38975, 5U, (const uint8_t *)"hello");
```

6. 将上传结果通过 `COM_DEBUG_LN()` 打印到 `USART1`
7. 进入空转 `while(1)`

这说明当前主程序重点不是完整“宠物定位业务闭环”，而是优先打通 `QS100` 上传链路。

## 7. 串口与接收机制

### 7.1 调试输出

- `printf` 已通过 `Com_Debug.c` 中的 `fputc()` 重定向到 `USART1`
- 若 `USART1` 没有任何输出，不要先怀疑串口助手，先检查：
  1. 程序是否跑到 `MX_USART1_UART_Init()`
  2. 是否在时钟 / RTC 初始化阶段提前进入 `Error_Handler()`
  3. `printf` 重定向是否被编译进当前工程

### 7.2 GPS 与 QS100 接收方式

- `USART2` 和 `USART3` 都使用 `HAL_UARTEx_ReceiveToIdle_IT()` 实现不定长接收
- `HAL_UARTEx_RxEventCallback()` 中按串口实例分发：
  - `USART2` -> `Int_AT6558R_CallBack()`
  - `USART3` -> `Int_QS100_ReceiveCallBack()`

这套机制能跑，但前提是：

- 中断已正确开启
- 回调内成功重新启动下一轮接收
- 响应缓存没有溢出

其中 `QS100` 接收链路已经是当前代码中最复杂、也最容易埋雷的部分。

## 9. 调试建议

### 9.1 串口无输出时

优先按以下顺序排查：

1. 板级供电、下载、复位是否正常
2. `HSE` 外部时钟是否真实可用
3. `SystemClock_Config()` 是否返回成功
4. `MX_RTC_Init()` 是否触发错误
5. `MX_USART1_UART_Init()` 是否执行
6. `printf` 重定向是否生效
7. 外部 USB 转串口链路是否正常

### 9.2 当前已确认事实

- 板子硬件基本已通过交叉验证排除
- 串口不输出更可能是工程初始化或重定向问题
- 若时钟或 RTC 失败，会在常规日志打印前进入死循环

### 9.3 调试策略

当前最稳妥的排查方法不是大改架构，而是先区分下面三类问题：

1. 程序未跑到 UART 初始化
2. 程序跑到了 UART，但 `printf` 重定向未生效
3. 程序已发送，但外部观察链路有问题

## 10. 当前风险与不足

- `main.c` 仍然是临时联调逻辑，服务器地址、端口和测试载荷被硬编码
- `App_Main` 业务流程尚未真正接入主循环
- 工程里存在部分中文注释编码异常，长期不处理会继续降低维护效率
- `Error_Handler()` 仍然是“关中断 + 死循环”，定位信息不足
- README 中描述的是当前仓库状态，不代表产品最终设计冻结

## 12. 后续建议

- 先把启动期的可观测性做扎实，确认时钟、RTC、串口初始化的真实执行路径
- 将 `main.c` 中的临时上传逻辑迁移到 `App/` 层，避免业务逻辑散落在入口文件
- 给 `QS100` 上传链路补充更明确的状态机和失败分级日志
- 在联调结束后，清理临时调试输出，保留必要但不过量的关键日志
