# E220x 核心机制

## 1. 驱动真正的“发动机”是什么

如果把 `Ebyte_E220x_SendPayload()`、`Ebyte_E220x_SetRx()` 看成“动作函数”，那它们背后的发动机其实是 5 个机制：

1. SPI 命令机制
2. 寄存器读写机制
3. FIFO 读写机制
4. 包参数配置机制
5. IRQ 驱动机制

理解了这 5 个机制，`ebyte_e220x.c` 基本就算吃透了。

## 2. SPI 命令机制

### 2.1 核心函数

- `Ebyte_E220x_WriteCommand()`
- `Ebyte_E220x_ReadCommand()`

### 2.2 它们在做什么

写命令时流程是：

1. `CheckReady()`
2. 拉低 CS
3. 发送命令字
4. 连续发送附加数据
5. 拉高 CS
6. 等 BUSY 结束

读命令时流程是：

1. `CheckReady()`
2. 拉低 CS
3. 发送命令字
4. 发送空字节作读前导
5. 连续读回数据
6. 拉高 CS
7. 等 BUSY 结束

### 2.3 为什么先抽这一层

因为大量高级功能都只是“不同命令 + 不同参数”的组合。先有命令原语，后面所有功能都变成参数封装问题。

这是一种非常典型、也非常值得模仿的驱动写法。

## 3. 寄存器机制

### 3.1 核心函数

- `Ebyte_E220x_WriteRegisters()`
- `Ebyte_E220x_WriteRegister()`
- `Ebyte_E220x_ReadRegisters()`
- `Ebyte_E220x_ReadRegister()`

### 3.2 设计思路

这里并没有让高层每次自己去拼地址高字节、地址低字节，而是把这类模板逻辑统一塞进寄存器原语里。

这有两个好处：

1. 避免重复代码
2. 避免地址拼装错误

### 3.3 为什么还保留单寄存器接口

因为单寄存器读写太常见了。例如：

- 写 XTA trim
- 写 XTB trim
- 写同步字高低字节

所以作者在批量寄存器原语之上又补了单寄存器快捷接口。这是很实用的 API 设计。

## 4. FIFO 机制

### 4.1 核心函数

- `Ebyte_E220x_WriteBuffer()`
- `Ebyte_E220x_ReadBuffer()`
- `Ebyte_E220x_SetPayload()`
- `Ebyte_E220x_GetPayload()`

### 4.2 分层关系

- `WriteBuffer` / `ReadBuffer`：纯 FIFO 原语
- `SetPayload` / `GetPayload`：面向“消息体”的一层包装

这种包一层的好处是，把“FIFO 地址偏移”这种设备细节从业务动作里拿走。

### 4.3 为什么 `GetPayload()` 先读长度再搬数据

因为无线包不是固定总长，接收侧必须先知道：

- 这次包多长
- 从 FIFO 哪个偏移开始读

所以先通过 `GetRxBufferStatus()` 获取长度和起始偏移，再读数据。这是对 FIFO 协议的忠实适配。

## 5. 包参数配置机制

### 5.1 LoRa 包参数

`Ebyte_E220x_SetLoraPacketParams()` 把以下参数封成一组：

- 前导码长度
- 可变长/定长包模式
- 负载长度
- CRC 开关
- IQ 反相

这就是一个很典型的“协议头模板构造函数”。

### 为什么值得学

因为作者没有把这些字段散落在发送流程里，而是集中成一个“参数块组装函数”。这让“包格式定义”变成了一个独立的概念。

### 5.2 FSK 包参数

`Ebyte_E220x_SetFskPacketParams()` 的处理方式与 LoRa 相同，但字段不同：

- 前导码位长度
- 前导码检测器
- 同步字长度
- 地址过滤
- 包长模式
- 负载长度
- CRC 类型
- Whitening

这再次说明作者的设计意识是：

- 不是“发数据”
- 而是“先定义包结构，再发数据”

### 5.3 调制参数配置

LoRa 用：

- `Ebyte_E220x_SetModulationLoraParams()`

FSK 用：

- `Ebyte_E220x_SetModulationFskParams()`

这是把“包结构”和“物理层调制参数”分开的一个好设计。很多初学者写驱动时会把这两种参数混成一坨，这里作者分得比较清楚。

## 6. 频率机制

### 6.1 核心点

频率不是直接把 Hz 写进去，而是要先换算成芯片要求的步进值。

源码用了：

- `XTAL_FREQ = 32000000`
- `FREQ_DIV = 33554432`
- `FREQ_STEP = XTAL_FREQ / FREQ_DIV`

然后：

```c
freq = (uint32e_t)((double)frequency / (double)FREQ_STEP);
```

这一步体现的是“工程参数 -> 芯片寄存器参数”的转换。

### 6.2 为什么前面还要校准

`Ebyte_E220x_SetRfFrequency()` 里在真正设频前做了：

- `Ebyte_E220x_Calibrate()`
- `Ebyte_E220x_CalibrateImage()`
- `SetStandby(STDBY_XOSC)`
- 写 trim 寄存器

这说明作者不是简单设寄存器，而是把“时钟和镜像校准”当成设频前置条件。这是硬件意识，而不只是软件动作。

## 7. IRQ 机制

### 7.1 核心函数

- `Ebyte_E220x_SetDioIrqParams()`
- `Ebyte_E220x_GetIrqStatus()`
- `Ebyte_E220x_ClearIrqStatus()`

### 7.2 设计思路

发送和接收都不是靠“盲等固定时间”，而是靠芯片内部 IRQ 状态位判断是否完成。

这说明作者采用的是“事件完成型驱动”，不是“拍脑袋延时型驱动”。

### 7.3 发送与接收如何用 IRQ

#### 发送

- 配置只关注 `IRQ_TX_DONE | IRQ_RX_TX_TIMEOUT`
- 发送后轮询 IRQ
- 回调发送结果

#### 接收

- 配置只关注 `IRQ_RX_DONE | IRQ_RX_TX_TIMEOUT`
- 轮询任务中判断 IRQ
- 收到包则搬运数据并回调

这种用法非常清晰：不同业务场景只关心相关事件，不会把所有 IRQ 一锅端。

## 8. 射频收发控制机制

### 8.1 外围电路控制

作者没有只操作芯片命令，还控制了外部 `TXEN` / `RXEN`：

- `Ebyte_E220x_RfIoEnterTxMode()`
- `Ebyte_E220x_RfIoEnterRxMode()`

这说明当前模块不是“芯片单体”，而是带外围收发开关的模块板级驱动。

### 8.2 为什么这一点重要

因为很多初学者会误以为“SPI 下命令 = 模块状态切换完成”。实际上对于这种模块，外部射频开关控制同样是流程的一部分。

作者把它显式拆出来，是对的。

## 9. 表驱动机制

源码里有两张很值得学的表：

### 9.1 LoRa 速率查表

`Ebyte_E220x_LoraBpsTable[8][3]`

它把：

- 空速档位
- 带宽
- 扩频因子
- 编码率

压成一个静态查表关系。

这意味着上层配置可以只暴露“档位”，不必每次都手工组合 SF/BW/CR。

### 9.2 FSK 带宽查表

`FskBandwidths[]`

它把带宽值与寄存器值映射起来。这说明作者知道“不是所有转换都适合写公式”，离散表驱动更稳。

## 10. 这套机制背后的设计意识

你真正要学的是下面这组意识：

1. 把芯片协议动作先抽成原语
2. 把原语再组合成业务动作
3. 把硬件前置条件编进流程里
4. 把离散配置做成查表
5. 用 IRQ 驱动完成判断，不靠瞎延时
