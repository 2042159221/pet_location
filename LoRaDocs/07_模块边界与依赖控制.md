# E220x 模块边界与依赖控制

## 1. 这份源码最值得学的，不是寄存器，而是边界

初学者最容易只盯着：

- 某条命令怎么发
- 某个结构体怎么写
- 某个指针怎么传

但真正拉开工程水平差距的，是你是否能画清楚模块边界。

当前这套源码，在边界设计上有明显的工程意识，也有几个真实的泄漏点。

## 2. 边界图

可以把它简化成下面这张关系图：

```text
App
  -> Inf_LoRa
    -> Ebyte_RF (统一接口对象)
      -> E220x Driver
        -> Port
        -> Callback
          -> Debug / 业务动作
```

旁边还有一个贯穿编译期的配置层：

```text
ebyte_conf.h
  -> 决定产品型号
  -> 决定 SPI CS 策略
  -> 决定发送检测方式
```

## 3. 每层“应该知道什么”

### 3.1 App 层

它应该知道：

- 什么时候初始化
- 什么时候发业务数据

它不应该知道：

- 模块寄存器
- SPI 协议
- IRQ 位定义

当前 `App_Main.c` 基本符合这个原则。

### 3.2 Inf 层

它应该知道：

- 本项目要把 LoRa 封成什么样的接口

它不应该知道：

- 具体是 E220 还是 E22
- HAL 细节

当前 `Inf_LoRa.c` 通过 `Ebyte_RF` 访问底层，说明这一层边界是成立的。

### 3.3 Core 层

它应该知道：

- 当前选择了哪种产品
- 如何把统一接口指到对应实现

它不应该知道：

- 业务数据是什么
- GPIO 怎么控制

当前 `ebyte_core.c` 做得很纯，只负责装配，不负责逻辑。

### 3.4 E220x Driver 层

它应该知道：

- 芯片命令
- 寄存器
- FIFO
- 调制参数
- 收发流程

它不应该知道：

- 上层业务数据语义
- GPS、JSON、服务器逻辑

当前它确实没有知道业务语义，这是好的。

### 3.5 Port 层

它应该知道：

- STM32 HAL
- SPI1
- 哪些 GPIO 接了 CS / RST / TXEN / RXEN / BUSY

它不应该知道：

- LoRa 参数
- 频率计算
- IRQ 语义

当前 `ebyte_port.c` 也基本符合。

### 3.6 Callback 层

它应该知道：

- 某次发送/接收结果要怎么处理

它可以知道：

- 打日志
- 发通知
- 切换回接收模式

当前它确实承担了“结果处理”角色。

## 4. 这套边界设计好的地方

### 4.1 产品型号选择被隔离了

`ebyte_core.c` 集中处理：

- E22
- E220
- E07
- E10
- E49

这避免了把型号判断污染到 `App` 或 `Inf`。

### 4.2 硬件平台差异被隔离了

`ebyte_port.c` 把 HAL SPI、GPIO、Delay 全吃掉了。以后换平台，理论上只要改这一层。

这就是典型的 BSP/Porting 层思路。

### 4.3 上层业务接口被进一步简化了

`Inf_LoRa.c` 的存在说明作者不满足于“有驱动就行”，还想给当前项目再做一层项目语义接口。

这是好习惯。

## 5. 这套边界设计泄漏的地方

### 5.1 芯片驱动直接依赖回调层

这是当前结构里最明显的泄漏点之一。

`ebyte_e220x.c` 在流程内部直接调用：

- `Ebyte_Port_TransmitCallback()`
- `Ebyte_Port_ReceiveCallback()`

这意味着：

- 驱动层不仅知道“有回调”
- 还知道“回调函数名是什么”

更理想的方式通常是：

- 在 `core` 或更上层注册回调函数指针
- 驱动层只调用已注册函数

当前做法胜在简单，但耦合确实更硬。

### 5.2 回调层又反过来调用统一接口层

`Ebyte_Port_TransmitCallback()` 里直接调用：

```c
Ebyte_RF.EnterReceiveMode(0);
```

这意味着回调层并不只是“通知层”，它还能驱动状态流转。

这是一个很现实的工程取舍：

- 好处：发完立刻回 RX，逻辑集中
- 坏处：回调不再是纯观察者，而是参与控制流程

### 5.3 头文件边界控制还不够严

当前多份头文件没有 include guard，而且不少内部函数没有 `static`。

这说明这套源码的边界控制更多依赖：

- 文件划分
- 头文件不暴露

而不是依赖 C 语言最严格的封装手段。

## 6. 上下层依赖是怎么被控制住的

虽然存在泄漏点，但整体还是能看出作者在控制依赖方向。

### 6.1 上层不碰 HAL

这是最重要的一条。否则一旦业务层直接调 GPIO/SPI，边界就彻底塌了。

### 6.2 上层不碰寄存器

`App` 和 `Inf` 都没有出现任何 `RADIO_` 命令常量，也没有出现寄存器地址。这说明协议细节被压住了。

### 6.3 Port 层不碰业务

`ebyte_port.c` 只做硬件动作，不解析状态码，不决定业务逻辑。

## 7. 如果你要把这份源码内化成自己的，该保留什么

建议保留：

1. `conf -> core -> driver -> port -> callback -> app` 这个总分层
2. `Ebyte_RF_t` 这种统一接口对象
3. 端口适配层和业务回调层的拆分
4. 底层原语层和高层动作层的拆分

## 8. 如果你要比它更进一步，应该改什么

建议增强：

1. 头文件补 include guard
2. 内部辅助函数尽量 `static`
3. 回调改成注册式，而不是硬编码函数名
4. 返回值体系更完整，不要大量 `while(1)`
5. 驱动内部状态与错误码设计成更清晰的结果对象
