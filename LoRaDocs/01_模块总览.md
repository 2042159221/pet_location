# E220x 模块总览

## 1. 文档范围

本文档只基于当前仓库中的实际源码展开，核心分析对象是以下文件：

- `Int/LoRa/ebyte_conf.h`
- `Int/LoRa/ebyte_core.h`
- `Int/LoRa/ebyte_core.c`
- `Int/LoRa/E220xMx/ebyte_e220x.h`
- `Int/LoRa/E220xMx/ebyte_e220x.c`
- `Int/LoRa/E220xMx/ebyte_port.h`
- `Int/LoRa/E220xMx/ebyte_port.c`
- `Int/LoRa/E220xMx/ebyte_callback.h`
- `Int/LoRa/E220xMx/ebyte_callback.c`
- `Inf/Inf_LoRa.h`
- `Inf/Inf_LoRa.c`
- `App/Src/App_Main.c`

不讨论芯片手册之外的推测，不把别的 EBYTE 项目经验硬塞进来。

## 2. 一句话先定性

这套源码不是“一个文件把 SPI 和 LoRa 全写完”的做法，而是明显分了层：

`产品选择配置层 -> 统一接口层 -> 芯片实现层 -> 硬件端口层 -> 回调扩展层 -> 项目接口层 -> 应用层`

这说明作者不是只想“把芯片跑起来”，而是想做成一套可在多个 EBYTE 型号之间复用的驱动骨架。

## 3. 目录与职责

| 层级 | 文件 | 主要职责 | 典型证据 |
| --- | --- | --- | --- |
| 配置层 | `Int/LoRa/ebyte_conf.h` | 选择具体产品、选择 SPI CS 控制方式、选择发送完成判定模式 | `EBYTE_E220_400M22S`、`EBYTE_PORT_SPI_CS_SOFTWARE`、`EBYTE_RF_TRANSMIT_CHECK_MODE` |
| 统一接口层 | `Int/LoRa/ebyte_core.h/.c` | 定义统一的 `Ebyte_RF_t` 接口，把不同芯片驱动装配成同一种“对象” | `Ebyte_RF_t` 结构体、`const Ebyte_RF_t Ebyte_RF` |
| 芯片实现层 | `Int/LoRa/E220xMx/ebyte_e220x.h/.c` | 负责 E220x 指令、寄存器、FIFO、LoRa/FSK 参数、收发流程、状态机 | `Ebyte_E220x_Init`、`Ebyte_E220x_SendPayload`、`Ebyte_E220x_SetRx` |
| 硬件端口层 | `Int/LoRa/E220xMx/ebyte_port.h/.c` | 把“驱动想做的硬件动作”翻译成 STM32 HAL 调用 | `HAL_SPI_TransmitReceive`、`HAL_GPIO_WritePin`、`HAL_Delay` |
| 回调扩展层 | `Int/LoRa/E220xMx/ebyte_callback.h/.c` | 把发送完成、接收完成、错误事件交给项目逻辑处理 | `Ebyte_Port_TransmitCallback`、`Ebyte_Port_ReceiveCallback` |
| 项目接口层 | `Inf/Inf_LoRa.h/.c` | 为当前项目再包一层更简单的 LoRa 接口 | `Inf_LoRa_Init`、`Inf_LoRa_SenData`、`Inf_LoRa_ReadData` |
| 应用层 | `App/Src/App_Main.c` | 决定什么时候初始化 LoRa，什么时候发送业务数据 | `App_Init`、`App_CollectAndUploadData` |

## 4. 分层调用关系

### 4.1 初始化链

1. `App_Init()` 调用 `Inf_LoRa_Init()`。
2. `Inf_LoRa_Init()` 先执行 `MX_SPI1_Init()`，再执行 `Ebyte_RF.Init()`。
3. `Ebyte_RF.Init()` 在当前配置下实际绑定到 `Ebyte_E220x_Init()`。
4. `Ebyte_E220x_Init()` 完成 E220x 复位、唤醒、模式设置、频率设置、功率设置，并最终进入接收模式。

### 4.2 发送链

1. `App_CollectAndUploadData()` 在 IoT 上报失败后调用 `Inf_LoRa_SenData()`。
2. `Inf_LoRa_SenData()` 调用 `Ebyte_RF.Send(data, len, 0)`。
3. 当前实际落到 `Ebyte_E220x_SendPayload(payload, size, timeout)`。
4. 芯片层完成包参数配置、FIFO 写入、切换 TX 模式、等待发送完成。
5. 发送完成后，调用 `Ebyte_Port_TransmitCallback()`。

### 4.3 接收链

1. `Inf_LoRa_ReadData()` 只是简单调用 `Ebyte_RF.StartPollTask()`。
2. 当前 `Ebyte_RF.StartPollTask()` 实际绑定到 `Ebyte_E220x_IntOrPollTask()`。
3. `Ebyte_E220x_IntOrPollTask()` 轮询 IRQ 状态，如果收到 `IRQ_RX_DONE`，则读状态、读 FIFO、调回调。
4. 回调函数是 `Ebyte_Port_ReceiveCallback()`。

## 5. 当前项目里的真实接线方式

这里必须说清楚，因为名字很容易误导你。

### 5.1 真正被驱动调用的回调

驱动芯片层直接调用的是：

- `Int/LoRa/E220xMx/ebyte_callback.c` 中的 `Ebyte_Port_TransmitCallback()`
- `Int/LoRa/E220xMx/ebyte_callback.c` 中的 `Ebyte_Port_ReceiveCallback()`

也就是说，当前工程里真正生效的“发送完成/接收完成”处理逻辑在 `ebyte_callback.c`，不是在 `Inf/Inf_LoRa.c`。

### 5.2 当前未接入真实调用链的函数

`Inf/Inf_LoRa.c` 里还有两个函数：

- `LoRa_ReceiveSuccessCallback()`
- `LoRa_TransmitSuccessCallback()`

从当前仓库检索结果看，这两个函数没有被驱动层调用，也没有别的地方引用。它们更像是项目作者曾经打算使用、但尚未真正接线的接口。

### 5.3 接收轮询入口当前也没有明显上层调度

`Inf_LoRa_ReadData()` 的存在说明当前接收模型依赖周期轮询 `Ebyte_E220x_IntOrPollTask()`。但从仓库调用关系看，`Inf_LoRa_ReadData()` 目前没有在 `App_Main.c` 中看到调用点。

这意味着：

- 发送路径已接通。
- 接收轮询入口已提供。
- 但应用层是否真的周期调用了接收轮询，目前从现有源码里看不到证据。

## 6. 为什么这种组织方式值得学

### 6.1 它把“可变的东西”隔开了

可变项主要有三类：

- 不同模块型号：E22、E220、E07、E10、E49
- 不同硬件平台：SPI、GPIO、Delay 的实现不同
- 不同业务逻辑：发送完成之后要做什么，接收到了数据要做什么

这三类变化分别被放到了：

- `ebyte_conf.h` + `ebyte_core.c`
- `ebyte_port.c`
- `ebyte_callback.c`

这不是“为了分层而分层”，而是因为这些点天然会变。

### 6.2 它不是纯面向对象，但明显借了“接口对象”的思想

`Ebyte_RF_t` 本质上就是一个函数指针表。C 语言里没有类和虚函数，作者就用结构体里的函数指针来做“统一接口”。

这让上层只需要写：

```c
Ebyte_RF.Init();
Ebyte_RF.Send(data, len, 0);
Ebyte_RF.StartPollTask();
```

而不用关心底层到底是 E220 还是 E22。

### 6.3 它把“通用 SPI 操作”与“射频业务动作”分开了

在 `ebyte_e220x.c` 里你能明显看到两种函数：

- 通用通信原语：`WriteCommand`、`ReadCommand`、`WriteRegisters`、`ReadRegisters`、`WriteBuffer`、`ReadBuffer`
- 业务动作：`SetRx`、`SetTx`、`SendPayload`、`Init`

这是一种非常值得学的分层方式。真正复用率高的往往是“原语层”，不是“业务流程层”。

## 7. 这套组织方式的边界感

### 7.1 控制暴露的手法

它主要靠两种方式控制暴露：

1. 通过头文件只公布一部分函数
2. 通过 `Ebyte_RF` 只给上层一个统一入口

例如 `ebyte_e220x.h` 对外暴露的主要是高层能力函数，而 `WriteRegisters`、`ReadBuffer` 这种底层工具函数虽然写成了外部链接函数，但并未放进公共头文件中给上层用。

### 7.2 但它的封装还不是“最严”

从 C 语言工程规范看，它还有两个明显的保守点：

1. 很多内部辅助函数没有声明成 `static`
2. 多个头文件没有做 include guard

这说明它的边界控制是“够用型”，不是“极致严谨型”。

## 8. 你读这套源码时应该带着什么问题

建议你读每一层时都问自己：

1. 这一层的输入是什么，输出是什么？
2. 这一层是否知道上层业务细节？
3. 这一层是否知道底层硬件细节？
4. 这一层改动时，会不会波及别层？
5. 这一层为什么要独立出来？

只要你一直按这 5 个问题读，源码就不会变成“只看函数细节，不懂系统结构”。
