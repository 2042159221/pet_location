# E220x 数据结构与指针设计

## 1. 为什么这一部分值得重点学

你关心的“底层 C 语言编写设计、结构体、指针设计”基本都集中在这里：

- 自定义整数类型
- 函数指针表
- 联合体 + 位域
- 嵌套结构体
- 静态全局状态
- `buffer + size` 指针接口

这些东西并不花哨，但非常接近真实 MCU 驱动的日常写法。

## 2. 自定义整数类型

`ebyte_port.h` 定义了：

- `uint8e_t`
- `uint16e_t`
- `uint32e_t`
- `int8e_t`
- `int16e_t`
- `int32e_t`

### 设计意图

显然是想统一整数位宽表达，避免直接写 `unsigned char`、`unsigned short` 这种原始类型。

### 值得学的点

- 代码可读性更好
- 看函数签名时一眼就知道位宽意图

### 也要看出问题

这里用的是：

```c
typedef unsigned long uint32e_t;
```

这在当前 STM32F1 工程和常见 ARM 编译器下通常没问题，但从可移植性看，不如直接使用 `<stdint.h>` 里的 `uint32_t` 稳。

所以你要学的是“位宽意识”，不是一定要照抄 `uint8e_t` 这种命名。

## 3. 函数指针结构体 `Ebyte_RF_t`

### 3.1 结构

`Ebyte_RF_t` 是一个典型的“接口对象”：

- `Init`
- `Send`
- `EnterSleepMode`
- `EnterReceiveMode`
- `StartPollTask`
- `InterruptTrigger`
- `GetName`
- `GetDriverVersion`

### 3.2 为什么用结构体装函数指针

因为 C 没有类，这种结构体就是“对象外壳”。

作者想表达的是：

- 不同芯片实现不同
- 但对上层来说，这些能力长得一样

所以用一个结构体把“能力集合”固定下来。

### 3.3 为什么这是好设计

它把“一个模块应该做什么”表达成了一个数据结构。

注意，这是非常重要的工程意识：

- 结构体不只是装数据
- 也可以装能力边界

## 4. 位域联合体 `RadioStatus_t`

### 4.1 结构

`RadioStatus_t` 是联合体，既可以整体看一个字节，也可以拆位看：

- `Reserved`
- `CmdStatus`
- `ChipMode`
- `CpuBusy`

### 4.2 为什么用 `union`

如果只用结构体位域，你很难同时保留“原始字节值”和“字段视图”。

用了 `union` 以后：

- `status.Value` 可以拿整字节
- `status.Fields.ChipMode` 可以拿某个字段

这个思路非常值得掌握。

## 5. 嵌套结构体 `PacketStatus_t`

### 5.1 结构特点

`PacketStatus_t` 先记录当前包类型 `packetType`，然后在 `Params` 里再分：

- `Gfsk`
- `LoRa`

每个子结构又有各自字段。

### 5.2 这类设计为什么好

因为 GFSK 和 LoRa 的包状态字段不同，但它们又属于同一个“包状态”概念。

作者没有粗暴地把所有字段摊平，而是用了：

- 一层公共语义：PacketStatus
- 一层按调制方式区分的具体字段

这是一种有边界感的数据建模方式。

## 6. 参数位联合体

源码里还有几类很典型的位域联合体：

- `CalibrationParams_t`
- `SleepParams_t`
- `RadioError_t`

这些结构反映了同一个模式：

1. 芯片寄存器字段本来就是按 bit 定义的
2. 软件里也按 bit 建模
3. 最后再把整个 `Value` 写出去或读回来

这种设计的价值是把“手册上的寄存器位图”变成“代码里的可读结构”。

## 7. 静态全局状态

`ebyte_e220x.c` 里有几组非常关键的静态变量：

- `OperatingMode`
- `PacketType`
- `PacketStatus`
- `FrequencyError`
- `RecvPacketBuffer`

### 7.1 这些变量的角色

#### `OperatingMode`

软件侧维护的模块状态影子。

#### `PacketType`

当前调制方式影子，用来决定后续参数写法和状态解释方式。

#### `PacketStatus`

最近一次接收包状态的缓存。

#### `FrequencyError`

名义上是频偏缓存，但在当前源码里只看到声明与读取，没有看到实际更新逻辑。

#### `RecvPacketBuffer`

驱动内部静态接收缓存。

### 7.2 为什么这里用静态全局

原因很现实：

- 这是单模块驱动
- 不需要多实例
- 避免频繁栈内分配大缓冲区
- 回调和轮询流程共享同一份状态

这对 MCU 场景是合理的。

### 7.3 代价是什么

- 不支持多实例
- 并发安全弱
- 重入性差

所以你要学的是：静态全局不是“错误”，而是明确建立在单实例前提上的取舍。

## 8. 指针设计模式

### 8.1 输入缓冲区指针

例如：

- `uint8e_t *payload`
- `uint8e_t *buffer`

这些参数本质上都表示“一段连续字节区域的起点”。

这是驱动里最基础、最常见的指针用途。

### 8.2 输出参数指针

例如：

- `uint8e_t *size`
- `uint8e_t *payloadLength`
- `uint8e_t *rxStartBufferPointer`
- `PacketStatus_t *pktStatus`

这类接口通过指针返回多个结果，避免返回复杂对象。

### 8.3 强制类型转换

例如：

```c
Ebyte_E220x_WriteCommand(RADIO_SET_STANDBY, (uint8e_t *)&standbyConfig, 1);
```

这里的意思是：

- `standbyConfig` 原本是枚举
- 底层命令发送函数只认字节流
- 所以把参数地址转成 `uint8e_t *`

这在设备驱动里很常见，因为最终发到总线上的就是字节。

### 8.4 这种写法为什么可行

因为这里发送的就是单字节参数，枚举底层值正好可以按字节解释。

但你也要意识到，这种写法依赖两个前提：

1. 参数对象的内存布局满足预期
2. 目标接口本身就是按字节协议定义

## 9. 查表结构体

`FskBandwidth_t` 是一个小而典型的结构体：

```c
typedef struct
{
    uint32e_t bandwidth;
    uint8e_t  RegValue;
} FskBandwidth_t;
```

它的作用不是装业务数据，而是装“映射关系”。

这是驱动里很值得学的结构体用途：

- 不是把逻辑埋在 `if-else`
- 而是把关系数据化

## 10. 这套指针与结构设计的优点

1. 简洁
2. 高效
3. 很贴近总线协议
4. 对 MCU 资源友好
5. 易于做查表和字段拼装

## 11. 这套设计的弱点

1. 很多输入指针其实可以加 `const`，但源码没有加
2. 绝大多数接口不做空指针检查
3. 静态全局导致不支持多实例
4. 位域写法可读，但不同编译器上的布局可移植性要谨慎

## 12. 你应该内化的不是“写法”，而是“判断标准”

以后你自己设计结构体和指针时，先问：

1. 这是在表达一段字节流，还是一个逻辑对象？
2. 这是单实例驱动，还是多实例驱动？
3. 这个结果是单值返回，还是多值回填？
4. 这个配置更适合查表，还是更适合公式？
5. 这个字段应该平铺，还是分层嵌套？
