# E220x 数据流分析

## 1. 先分清两种流

学习驱动时，最容易把两种流混在一起：

- 数据流：字节数据怎么从上层进来，再进入 FIFO，再从 FIFO 出来
- 控制流：程序如何决定“现在该发、该收、该等待、该回调”

这份源码里，两种流是交织的，但仍然能拆开来看。

## 2. 初始化流

### 2.1 入口

入口在：

- `App/Src/App_Main.c` 的 `App_Init()`
- `Inf/Inf_LoRa.c` 的 `Inf_LoRa_Init()`

### 2.2 实际流程

1. `App_Init()` 调用 `Inf_LoRa_Init()`
2. `Inf_LoRa_Init()` 初始化 `SPI1`
3. `Inf_LoRa_Init()` 调用 `Ebyte_RF.Init()`
4. `Ebyte_RF.Init()` 在当前产品配置下映射到 `Ebyte_E220x_Init()`
5. `Ebyte_E220x_Init()` 做以下事情：
   - 复位模块
   - 通过 CS 拉低触发唤醒
   - 设软件状态为 `MODE_STDBY_RC`
   - 切到 `STDBY_XOSC`
   - 修正 XTA/XTB trim
   - 开启 DIO2 自动控制射频开关
   - 选择内部稳压模式
   - 复位 FIFO 基址
   - 配置调制方式
   - 配置空速 / 扩频因子 / 带宽 / 编码率
   - 配置同步字
   - 配置载波频率
   - 配置发射功率
   - 最后进入接收模式

### 2.3 初始化的数据输入

初始化并不是“纯代码逻辑”，它实际上依赖几组静态配置：

- `ebyte_conf.h` 决定产品型号、SPI CS 策略、发送完成检测模式
- `ebyte_e220x.h` 决定频点、功率、LoRa/FSK 模式、空速、同步字等
- `ebyte_port.c` 决定具体如何访问 MCU 硬件

也就是说，`Init()` 的“输入”不只是参数，更多来自编译期宏和硬件端口实现。

## 3. 发送数据流

### 3.1 业务起点

`App_CollectAndUploadData()` 先整理 GPS、步数，并把数据转成 JSON 字符串。然后：

1. 优先尝试 QS100 上传
2. 失败后退回 LoRa
3. 调用 `Inf_LoRa_SenData((uint8_t *)g_upload_data.json_data, strlen(g_upload_data.json_data))`

这说明 LoRa 在当前项目里承担的是“退路链路”。

### 3.2 接口层传递

`Inf_LoRa_SenData()` 没做任何业务加工，直接调用：

```c
Ebyte_RF.Send(data, len, 0);
```

这里的三个参数非常典型：

- `data`：数据首地址
- `len`：有效长度
- `0`：发送超时参数

这是标准的“缓冲区 + 长度 + 控制参数”接口风格。

### 3.3 芯片层发送流程

`Ebyte_E220x_SendPayload()` 的流程非常规整：

1. 根据 `RF_TX_RX_MODE` 选择配置 LoRa 包参数还是 FSK 包参数
2. 调用 `Ebyte_E220x_SetPayload()` 把数据写进模块 FIFO
3. 调用 `Ebyte_E220x_SetTx(timeout)` 进入发送模式
4. 如果 `EBYTE_RF_TRANSMIT_CHECK_MODE == 1`
   - 阻塞轮询 IRQ
   - 直到读到非 0 中断状态
   - 清除 IRQ
   - 调发送完成回调

### 3.4 发送时的数据落地路径

从“字节视角”看，发送数据这样走：

1. `App_Main.c` 中的 JSON 字符串缓冲区
2. `Inf_LoRa_SenData()` 形参 `data[]`
3. `Ebyte_RF.Send()` 形参 `buffer`
4. `Ebyte_E220x_SendPayload()` 形参 `payload`
5. `Ebyte_E220x_SetPayload(payload, size)`
6. `Ebyte_E220x_WriteBuffer(0x00, payload, size)`
7. `Ebyte_Port_SpiTransmitAndReceivce()` 按字节写入 E220x FIFO

也就是说，发送链里几乎没有额外拷贝。核心设计意图就是：

- 上层提供一块连续内存
- 下层按长度顺序写出
- 不做复杂缓存管理

### 3.5 发送完成后的控制流

当前源码里发送完成后不会停留在 TX，而是：

1. `Ebyte_E220x_SendPayload()` 调用 `Ebyte_Port_TransmitCallback()`
2. `Ebyte_Port_TransmitCallback()` 内部第一件事就是 `Ebyte_RF.EnterReceiveMode(0)`

这说明作者默认的工作模式是：

- 平时在 RX
- 要发时短暂切到 TX
- 发完立刻回 RX

这很符合半双工无线模块的典型使用方式。

## 4. 接收数据流

### 4.1 入口不是中断函数，而是轮询任务

尽管驱动里有 IRQ 配置，但当前工程真正暴露给上层的是：

```c
void Inf_LoRa_ReadData(void)
{
    Ebyte_RF.StartPollTask();
}
```

也就是“周期轮询任务”模式，而不是“MCU 外部中断直接接管”模式。

### 4.2 接收流程

`Ebyte_E220x_IntOrPollTask()` 的流程是：

1. 读 IRQ 状态
2. 如果 IRQ 为 0，什么都不做
3. 如果 IRQ 非 0，先清除 IRQ
4. 如果包含 `IRQ_RX_DONE`
   - 获取包状态
   - 从模块 FIFO 读出数据到 `RecvPacketBuffer`
   - 调接收回调
   - 重新进入 RX
5. 否则
   - 把当前状态和空长度也交给回调处理

### 4.3 接收时的数据落地路径

从“字节视角”看，接收数据这样走：

1. E220x 芯片内部 FIFO
2. `Ebyte_E220x_GetRxBufferStatus()` 得到长度和起始偏移
3. `Ebyte_E220x_ReadBuffer(offset, buffer, size)` 从 FIFO 读出
4. 目标缓存是芯片层静态数组 `RecvPacketBuffer`
5. `Ebyte_Port_ReceiveCallback(state, RecvPacketBuffer, recvLength)` 把指针传给回调

关键点在第 4 步和第 5 步：

- 接收缓存不是在回调里现分配的
- 它是驱动内部静态缓存
- 回调拿到的是这块静态缓存的首地址

这意味着回调层如果要长期保存数据，应该自己再复制一份，而不是把这个指针长期保留。

### 4.4 接收流程里的长度控制

`Ebyte_E220x_GetPayload()` 先获取长度，再比较 `*size > maxSize`。如果越界则返回 `1`。

设计意图很明确：

- 由底层统一做长度门槛检查
- 避免直接把过长数据写爆缓存

但当前源码里 `Ebyte_E220x_IntOrPollTask()` 并没有检查 `Ebyte_E220x_GetPayload()` 的返回值，而是直接继续回调。这属于“有保护意识，但落实不完整”。

## 5. 控制流与数据流如何交汇

最值得你学的不是某个单独函数，而是“数据流”和“控制流”如何合在一起：

### 5.1 发送场景

- 数据流：业务缓冲区 -> FIFO
- 控制流：配置包参数 -> 切 TX -> 轮询 IRQ -> 清 IRQ -> 回调 -> 回 RX

### 5.2 接收场景

- 控制流：轮询 IRQ -> 判断 `IRQ_RX_DONE`
- 数据流：FIFO -> `RecvPacketBuffer`
- 控制流：回调 -> 重新进入 RX

这就是典型的事件驱动式嵌入式驱动：

- 平时不处理字节
- 有事件再决定是否搬运数据

## 6. 参数是怎么从需求落到代码的

从需求角度看，作者在做的是三件事：

### 6.1 需求一：模块必须能切收发

落地成：

- `Ebyte_E220x_RfIoEnterTxMode()`
- `Ebyte_E220x_RfIoEnterRxMode()`
- `Ebyte_E220x_SetTx()`
- `Ebyte_E220x_SetRx()`

### 6.2 需求二：上层只想给一包数据，不想管寄存器

落地成：

- `Ebyte_RF.Send()`
- `Ebyte_E220x_SendPayload()`
- `Ebyte_E220x_SetPayload()`

### 6.3 需求三：接收完成后要把数据抛给业务层

落地成：

- `Ebyte_E220x_IntOrPollTask()`
- `Ebyte_Port_ReceiveCallback()`

这就是“从需求到代码”的核心过程：

先识别稳定动作，再把动作封成 API，然后让 API 去驱动底层寄存器。

## 7. 你读源码时最该盯住的 4 个数据节点

1. 业务输入缓冲区
2. 芯片内部 FIFO
3. 驱动内部静态接收缓冲区 `RecvPacketBuffer`
4. 回调函数形参里的 `buffer` 与 `length`

只要你抓住这 4 个节点，再配合 IRQ 状态去看，就很容易理清“字节到底在哪里”。
